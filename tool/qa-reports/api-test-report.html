Methodology & Scope

Test levels applied

  • Contract verification — every call‑site read as a smoke test: request shape, response handling, error taxonomy, idempotency.
  • Security‑rules simulation — every allowed/denied path traced from the Dart call site through firestore.rules, then the rule re‑evaluated against a crafted‑client threat model (an attacker with only the APK and a valid user).
  • Data‑integrity analysis — write order, transactional boundaries, counter reconciliation, race windows.
  • Resilience — offline, network‑request‑failed, rate‑limit, too‑many‑requests, and storage outages.
  • Index planner — each query matched against declared indexes to verify it runs without a "needs index" error.

Out of scope

  • Live execution against the prod Firebase project (would require service‑account or tester credentials; held back per project convention to pause on credentials).
  • Penetration testing (no credential brute force, no fuzzing of Cloudinary).
  • UI/UX screenshot regression — belongs in the forthcoming end‑to‑end phase.

Key Findings (25)

FND-01 — Real Firebase service-account private key shipped in APK Critical

Surface: FCM / Identity Toolkit File: assets/fcm-service-account.json Seen by: NO-08, US-18

The release APK bundles a real Google service-account JSON with a private_key granting firebase.messaging and identitytoolkit scopes. Any attacker who extracts the APK's assets (a 5-second unzip) can: (a) send arbitrary push notifications to every member, (b) disable or re-enable any user via Identity Toolkit, and (c) potentially escalate depending on the IAM role attached to that SA on agaram-15347.

This is explicitly against Firebase's client-app guidance. Free-tier constraint doesn't change it — it's a credential exposure, not a quota issue.

Remediation
  1. Immediately rotate the service account in Google Cloud IAM → delete the leaked key.
  2. Strip the asset from the build: revert fcmServiceAccountAsset to the placeholder shape, remove the file from assets/, and gate admin FCM send behind a trusted path.
  3. Since the project is Spark (no Cloud Functions), move FCM send to a free alternative trusted runner: Cloudflare Workers free tier or EAS Background Job or a tiny Vercel serverless function that proxies the SA. App calls POST /push authenticated with the user's Firebase ID token; the proxy verifies the admin claim and forwards to FCM.
  4. Alternatively, for purely topic-based pushes, move to Firebase Cloud Messaging's legacy server key stored remotely (still better than in-APK) or use FCM via Admin SDK on a trusted backend.

FND-02 — Members can edit their own stars, email, joinedAt Critical

Surface: Firestore rules / usersFile: firestore.rules:49-54Seen by: US-08, US-09, US-10

The self-update branch only blocks role, isPresident, position, active. That means every other field is writable: stars, email, phone, photoUrl, joinedAt, lastReadNotificationsAt. A member using a crafted Firestore client can give themselves 10,000 stars and top the leaderboard. Audit-trail fields (joinedAt) are also forgeable.

Remediation (one-line rule change) 
allow update: if isSignedIn() && (
  (request.auth.uid == uid
    && onlyChanged(['name','phone','photoUrl','lastReadNotificationsAt']))
  || (isAdmin() && notChanged(['isPresident']))
  || isPresident()
);

Note the inversion: move from a denylist (notChanged) to an allowlist (onlyChanged). The current approveTask/attendance transactions update stars under the admin branch (admin updates another user's doc), so they are unaffected. The member's own client never needs to write stars.

FND-04 — Attendance is forgeable without a QR scan Critical

Surface: Firestore rules / attendanceFile: firestore.rules:94-103Seen by: AT-09, AT-10

The rule only checks memberUid == request.auth.uid on create. It does not validate against the event's qrSecret, the 3-hour window, or the method/starsAwarded fields. A member with a crafted client can write /events/X/attendance/{theirUid} at any time with starsAwarded: 999. Combined with FND-02, the attacker also credits themselves the stars.

Remediation

Because Spark rules can only get() the parent event doc, we can enforce the secret server-side if the member is required to include the secret in the payload:

match /attendance/{memberUid} {
  allow create: if isSignedIn()
    && memberUid == request.auth.uid
    && request.resource.data.method in ['qr','manual']
    && request.resource.data.starsAwarded == 2
    && request.resource.data.userId == request.auth.uid
    && (
      // QR path: caller must present the current secret.
      (request.resource.data.method == 'qr'
        && request.resource.data.qrSecretUsed ==
             get(/databases/$(database)/documents/events/$(eventId)).data.qrSecret
      )
      || isAdmin() // manual path stays admin-only
    );
}

Then update AttendanceService.checkInWithQr to write qrSecretUsed: payload.secret into the doc. The read-your-own-event cost is 1 extra document read per check-in — acceptable on free tier.

FND-03 — Members can forge review metadata on task submit High

File: firestore.rules:82-91Seen by: EV-17

reviewNote, reviewedBy, reviewedAt are present in the member-allowed onlyChanged list. The code today writes them as nulls during resubmission, but a crafted client can set them to arbitrary values. Admin approval will eventually overwrite, but in the interim the review queue displays forged metadata.

Remediation 
&& onlyChanged([
    'proofUrl', 'proofType', 'memberNote',
    'status', 'submittedAt'
])
&& request.resource.data.reviewNote == null
&& request.resource.data.reviewedBy == null
&& request.resource.data.reviewedAt == null

Explicit null-assertion keeps resubmission working while denying spoof values. Also update EventService.submitProof to unset instead of explicit null if you prefer: FieldValue.delete().

FND-05 — Deactivated users keep their active session High

File: lib/core/auth_service.dart:62-76Seen by: AU-18

The app reads active once on sign-in and never again. An admin who deactivates a currently signed-in user does NOT kick them out. Identity Toolkit disable also doesn't invalidate existing ID tokens immediately — Firebase caches tokens for up to 1 hour.

Remediation
  • Subscribe to the current user's users/{uid} doc. On active:false → signOut().
  • Add a Firestore-rule guard on all writes: && userDoc(request.auth.uid).data.active == true. Costs one extra get per write, but on Spark with < 500 daily users this is fine.

FND-06 — Event delete leaves orphan subcollections High

File: lib/core/event_service.dart (no delete implementation shown)Seen by: EV-04

Firestore doc.delete() on the event parent doesn't recursively delete tasks, attendance, gallery, wallet. Those docs remain and still surface in collection-group queries. A member's My Tasks will show tasks for a deleted event, crashing nav.

Remediation

Spark has no Cloud Functions — do it client-side in EventService.deleteEvent: iterate each subcollection, batch-delete in chunks of 500. Wrap in a progress UI. Alternatively, soft-delete (archived: true) and filter everywhere.

FND-07 — Counters (tasksCount, walletCounts) drift on failure High

File: lib/core/event_service.dart:88-91, lib/core/wallet_service.dart:49-58Seen by: EV-11, WA-05

Task creation: doc add then a separate event update with FieldValue.increment(1). If the second fails (rules race, offline), the task exists but count is stale forever. Same shape for wallet.

Remediation

Wrap in a runTransaction, or embrace eventual consistency: don't store counters. Read with .count() aggregate query when you need them. For Spark, an aggregate query is 1 read regardless of size — cheaper than the current design which still costs N reads to list.

FND-18 — Gallery & Wallet create path accepts forged fields High

File: firestore.rules:105-123Seen by: GA-02, WA-06

allow create: if isSignedIn(); — no validation. A crafted client can set uploadedBy to another user's uid, sizeBytes to a lie, or put a non-Cloudinary URL in url.

Remediation 
allow create: if isSignedIn()
  && request.resource.data.uploadedBy == request.auth.uid
  && request.resource.data.url.matches('https://res\\.cloudinary\\.com/dttox49ht/.*');

FND-08 — Password reset error copy can leak via default branch Medium

File: lib/core/auth_service.dart:101-118

The default branch returns e.message raw. In some SDK versions this could include "No user found for email..." allowing enumeration on the reset form.

Remediation

Collapse unmapped codes to a fixed "If that email exists, we've sent a reset link." Log the underlying code for debugging only.

FND-09 — FCM topic reconciliation on role change Medium

Seen by: AU-15

If an admin demotes themselves or is demoted by the president mid-session, they remain subscribed to admins_only until next sign-in.

Remediation

Watch the current user's role field; on change, call FcmService.unsubscribeAll + subscribeForMember/Admin.

FND-10 — Member creation can orphan Auth account on process kill Medium

File: lib/core/members_service.dart:65-92

Between createUserWithEmailAndPassword and the Firestore set, if the app is killed the Auth account exists but has no profile. On next sign-in attempt, _loadUser shows "not set up yet" — stuck state.

Remediation

Write Firestore first with a temporary placeholder uid, OR on the next admin open of Members, reconcile by listing Auth accounts (not possible on Spark) — so at least surface a "retry creation" button that deletes the Auth account if the Firestore write fails.

FND-11 — Deactivate flow is non-atomic between Firestore and Identity Toolkit Medium

File: lib/core/members_service.dart:108-116

If Identity Toolkit call fails, Firestore is already flipped — user is "deactivated" in UI but Auth still accepts sign-in. Sign-in defense-in-depth (active:false check) handles it, but admin UI will be misleading.

Remediation

Flip Identity Toolkit first, Firestore second — that way, a failure leaves Auth allowing sign-in (correct) and the UI shows still-active (consistent). Add retry button.

FND-12 — President-deleted user leaves orphan Auth account Medium

Seen by: US-19

Firestore user doc is deletable but Firebase Auth has no client SDK delete (and no Spark Cloud Function). The Auth account persists forever, consuming a "user" slot and still capable of sign-in (though it'll hit "not set up yet").

Remediation

Before Firestore delete, disable the Auth account via Identity Toolkit (reusing _setAuthAccountDisabled). Document that full Auth-account deletion requires a backend.

FND-13 — Upload size caps inconsistent across surfaces Medium

Seen by: EV-09, GA-08, WA-09

Only task proofs are size-capped (10 MB). Gallery, wallet, banners, avatars have no client enforcement. Cloudinary free plan's 10 MB hard limit will throw opaque errors for users.

Remediation

Centralise caps in AppConfig (e.g. maxImageMb=10, maxPdfMb=10, maxBannerMb=5). Enforce before Cloudinary call; show friendly copy.

FND-14 — rejectTask is not transactional Medium

File: lib/core/event_service.dart:194-235

Approve uses a transaction, reject doesn't. Two simultaneous rejects → last-writer-wins; the follow-up .get() for push body may not match the written doc.

Remediation

Wrap in runTransaction; capture the assignee uid and title inside the txn; push outside.

Intentional, for the My Tasks/Review Queue UX. Worth a privacy note: a curious member can enumerate every task doc (including proofUrls, memberNotes, reviewerNotes). If tasks can contain sensitive content, tighten: allow CG read only when assignedTo == uid OR admin.

FND-16 — Device-clock trust in QR window Low

UX only (no security impact — client-only rule). Accept; optionally use Firebase server timestamp roundtrip to warn "your device clock looks wrong."

FND-19 — Cloudinary assets orphaned after Firestore delete Low

Expected on Spark (no signed admin API integration). Track as product debt; plan a monthly admin "cleanup" job or move to signed Cloudinary + backend later.

FND-21 — Silent FCM failures Low

_fireAndForget hides errors in release builds. Add a lightweight counter in a Firestore admin-only pushMetrics doc (one write per day with daily failure count) to notice systemic issues without a backend.

FND-22 — Unread-notification count is wrong above 50 Medium

File: lib/core/notifications_service.dart:46-59

Firestore inequalities auto-order ASC on the same field. Combined with limit(50), the code returns the oldest 50 unread. A member with 120 unread sees a badge stuck at 50.

Remediation

Use .count() aggregate instead of .get(): await query.count().get(). On Spark this is 1 read regardless of matching count. Remove the limit(50).

FND-23 — Unread-count stream re-runs on every user-doc change Low

Stars increments cause the notifications query to re-fire. Decouple: stream user doc just for lastReadNotificationsAt, keep it distinct.

FND-24 — Hardcoded April-2026 theme fallback Low

Will feel dated quickly. Replace with a neutral "Theme coming soon" or compute a generic tamil/english pair from the month name.

FND-25 — Unsigned Cloudinary preset is open for quota abuse Low

Anyone with the APK (or the cloud/preset combo) can POST to Cloudinary and burn your 25 GB/month quota. Tighten the preset in Cloudinary dashboard: Allowed formats = image + pdf, Max file size = 10485760, Use filename off, Overwrite off, enable Moderation if sensitive.

Performance & Reliability Observations

  • Read budget (Spark: 50K / day) — estimated average per user per day under normal use: ~40 reads (home screen warm-up 10, events list 6-12, My Tasks CG 3-6, notifications unread stream 1-2 on user-doc changes, leaderboard on demand 50). With 200 active members, 8,000 reads/day — well under budget until you cross ~1,000 users.
  • Write budget (Spark: 20K / day) — dominated by attendance check-ins (2 writes per check-in — attendance doc + user stars), task submits/approves, gallery/wallet. 200 members × 2 check-ins/week = 400 writes/week; fine.
  • Transaction contention — the users/{uid} doc is the hot path. Leaderboard updates, approvals, attendance all touch it. Firestore transactions can retry up to 5× on contention. Risk is low at club scale; document if you ever multi-admin-approve in batches.
  • Cold-start FCM handler — getInitialMessage fires before the first frame; navigator may not be ready. Code guards with AppNav.navigator == null but may drop a tap early. Consider buffering the payload until navigator mounts.
  • ReminderService.syncUpcoming runs on every auth change + every app resume; 30-day lookahead = ~N reads. Acceptable, but cache the last-sync time and skip if < 5 min since last run.
  • Image caching — cached_network_image does client-side caching; good. For leaderboard avatars, consider memCacheWidth to reduce memory.

Recommendations — Blocking for Release

  1. Rotate the leaked service account now, strip assets/fcm-service-account.json from the build, and move FCM send off-device. (FND-01)
  2. Invert the user self-update rule to an allowlist. Single-line fix. (FND-02)
  3. Harden attendance create rule to require a matching qrSecretUsed or admin. (FND-04)
  4. Lock down task-submit field allowlist so members can't forge review metadata. (FND-03)
  5. Watch active in session so deactivated users lose their live session. (FND-05)
  6. Decide orphan-subcollection policy on event delete: hard cascade delete or soft-archive. (FND-06)
  7. Replace drift-prone counters with aggregate queries or make writes transactional. (FND-07)
  8. Harden Cloudinary unsigned preset (file-size & format limits) and add app-side size caps to every upload path. (FND-13, FND-25)
  9. Fix unread-count to use .count(). (FND-22)

Items 1–4 are release-blocking. Items 5–7 are release-blocking only if the club's data already has real value; otherwise next sprint. Items 8–9 are polish.
